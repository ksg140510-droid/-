   - DinoCapture가 항상 실행 중이어야 함 (카메라 계속 물고 있어야 하므로 카운터 앱은 "창 캡처 모드"로 전환 필수 — 직접 카메라 모드와 동시 사용 불가)
   - DinoCapture 창이 최소화되면 캡처 안 됨(화면에 실제로 그려져 있어야 함)
   - 다른 창에 가려지면 그 창 내용이 캡처됨
   - 창 크기가 바뀌면 배율 표시 위치가 툴바 줄바꿈 때문에 흔들림 (고정 크기로 강제 리사이즈하는 코드까지 짜다가 중단)
   - → **사용자 판단: 너무 번거로움, 실전에서 못 씀.** 관련 코드 전부 제거함(pytesseract import, `_read_dino_magnification_ocr`, `_start_dino_ocr_watcher`, `_find_dino_window_handle`, `_ensure_dino_window_size` 등).
3. **디노라이트 직접 카메라 공유(카메라 두 앱이 동시에 열기)** — 테스트 결과 하드 실패(DinoCapture가 카메라 잡고 있으면 다른 프로세스는 아예 못 엶). 가상카메라 분배 소프트웨어도 검토했으나, 그러면 DinoCapture가 진짜 장치를 안 보게 되어 AMR 자체가 깨질 위험 있어서 포기.

### 2026-07-15 밤, 실제 장비로 직접 검증한 결과 (다음 세션 시작점)
DNX64.dll을 직접 로드해서 우리 자체 cv2 비디오 스트림을 켜놓은 상태로 하나씩 호출 테스트함(카운터 앱은 항상 자기 카메라 스트림을 열어두므로 실전 조건과 동일).

**확인됨 — 실제로 동작함 (구현 추천)**
- `GetAutoExposure` / `SetAutoExposure(idx, 0)` — 끄기 전 1 → 끈 후 0으로 readback 정상 변경 확인. **자동노출 끄기 성공.**
- `GetExposureValue` / `SetExposureValue` — Set 후 readback 값이 실제로 바뀜(864 → 4091, -5 스탑 설정 시). **노출 고정 성공.**
- `GetDeviceIDA`, `GetConfig` — 정상 동작.

**호출은 성공하지만 효과 미확인/의심**
- `GetGain` / `SetGain` — Set 호출은 에러 없이 실행되지만, 스트림 활성 상태에서도 readback이 항상 0으로 그대로임 → 이 장비에서는 게인 제어가 실질적으로 안 먹는 것으로 보임(또는 별도 전제조건 필요, 추가조사 필요).
- `GetISO` / `SetISO` — 위와 동일하게 readback 불변, 미확인.
- `SetLEDState` — 호출 자체는 에러 없음. 실제 LED가 꺼졌다 켜지는지는 테스트 중 육안 확인 못함(다음 세션에서 재확인 필요).
- `GetSerialNum` — 호출 성공하지만 빈 문자열 반환(이 장비에 시리얼이 안 박혀있는 듯) → 장비 추적용으로는 이 특정 유닛에서는 못 씀.

**중요 발견**: AMR과 달리 노출/자동노출 제어는 **DinoCapture 없이 우리 자체 cv2 스트림만 열어도 정상 동작**함. 즉 카운터 앱이 평소처럼 카메라를 직접 열어놓은 상태 그대로 노출 고정 기능을 넣을 수 있음 — 추가 아키텍처 변경 불필요.

**다음 세션에서 바로 시작할 것**: `SetAutoExposure(idx, 0)` + `SetExposureValue(idx, 고정값)`을 카운터 앱 시작 시(또는 캘리브레이션 시작 시) 자동으로 걸어주는 기능 구현. 적정 고정 노출값은 실제 조명 조건 보고 잡아야 함(오늘 테스트는 임의로 -5만 넣어봄, 실사용 값 아님).

3. DNX64가 원인인가 테스트 — **DNX64를 아예 안 건드린 실행(설정파일 locked_brightness/contrast를 null로 비움)에서도 재현됨 → DNX64는 원인이 아님을 확인.**
4. `StopMonitoring()` 호출 추가 — 효과 없음(DNX64가 원인이 아니었으니 당연함)
5. **진짜 원인 발견**: `_cam_loop` 스레드가 `cam.read()`를 쉴 새 없이 반복 호출하는 구조라, 종료 순간에 OpenCV/DirectShow 네이티브 호출 도중일 확률이 매우 높음. 그 상태에서 바로 `os._exit(0)`으로 죽이면 Windows가 그 커널 I/O가 끝날 때까지 프로세스 정리를 미루는 것으로 추정.
6. **해결책**: `_on_close`에서 `self.running = False` 신호를 준 뒤 **`time.sleep(0.4)`로 카메라 스레드가 스스로 루프를 빠져나와 `cam.release()`할 시간을 준 다음** 강제종료 진행. → **python 직접 실행 2회, 빌드된 exe 2회, 총 4회 연속 깨끗하게 종료 확인됨.** 실사용 패턴(카메라 화면 보다가 종료)으로도 확인됨.

### 다크/라이트 테마 전환 시 버튼이 형광색으로 남는 버그 — 원인 특정 + 해결됨
- **근본 원인**: `_D2L`(다크→라이트 색상표)에서 여러 다크색이 같은 라이트색으로 겹쳐 매핑돼 있음(예: `#1a5c2e`/`#00c853`/`#2e8b57`/`#aaffaa` 전부 `#1b5e20`으로). `_L2D`(역방향)는 `{v:k for k,v in _D2L.items()}`로 자동 생성되는데, 이렇게 여러 키가 같은 값으로 겹치면 파이썬 딕셔너리 특성상 **마지막에 정의된 것만 살아남음** → 라이트→다크로 되돌아갈 때 전혀 다른(더 쨍한) 색으로 복원되는 버그.
- 이 버그는 `_apply_dark_button_styles` 같은 명시적 재색칠 함수가 있는 버튼(`btn_save`, `btn_history`, `_cam_save_btn`, `_cap_btn`, 새로 만든 LED/노출고정/화질고정 버튼 등)에도 영향을 줬음 — 왜냐하면 `_retheme()`가 이 버튼들도 먼저 잘못 칠하고 지나가는데, `_cam_save_btn`/`_cap_btn`처럼 `_apply_icons`에서만 재보정되는 버튼은 타이밍/누락으로 문제가 남을 수 있었음.
- **해결책**: 컬러맵 전체를 다시 설계하는 대신(리스크 큼), **명시적으로 색을 관리하는 버튼들을 `_retheme()`의 자동 변환 대상에서 아예 제외**(`_RETHEME_EXCLUDE_ATTRS` 목록 추가). 이 버튼들은 항상 각자의 명시적 함수가 마지막에 정확한 색을 칠하므로 자동 변환이 필요 없음.
- 사용자가 다크→라이트→다크 왕복 후 "이력 조회", "화질 고정", "저장" 버튼 모두 정상 확인함.
